# プログラミング初心者が実際に困ったこと2つ：GitHub迷子事件とClaude API制限問題の解決法

こんにちは。プログラミング開発2日目の私が、本日経験した2つの大きな課題とその解決策について、同じように困っている初心者の皆さんに向けて書きました。

---

## 導入：初心者あるあるの実体験

プログラミングを始めたばかりの頃、想像以上にたくさんの「わからない」に直面します。

私も本日、朝から夜まで以下の2つの問題に悩まされました：

1. **GitHub操作がわからず、作成したページがどこにあるのか見つけられない**
2. **Claude AIの使用制限にかかり、作業が全く進まない**

「こんなに困るの？」と思う人もいるかもしれません。でも、開発初心者にとっては本当に深刻な問題です。

本記事では、これら2つの課題にどう向き合い、どう解決したのかをお伝えします。

## 第1章：GitHub操作ガイド〜ページがどこにあるか分からない問題

### 問題：「あちこち探してやっと見つけた」

GitHub上でファイルを作成・更新したのに、その成果物がどこにあるのかわからない。

これが私が本日経験した第一の課題です。リポジトリはできたけど、Pages設定？デプロイされたページ？URLはどこ？そんな状態でした。

### 解決ステップ：具体的な操作方法

#### ステップ1：GitHubのリポジトリページを開く

まずは、あなたのリポジトリページにアクセスします。URLは以下の形式です：

```
https://github.com/[ユーザー名]/[リポジトリ名]
```

#### ステップ2：Settings（設定）タブを探す

リポジトリのページを開いたら、上部のタブを見てください。

- Code
- Pull requests
- **Settings** ← ここをクリック

#### ステップ3：Pages設定を確認する

Settings内の左側メニューから「Pages」を探します。

```
左側メニュー
├── General
├── Collaborators
├── Data and privacy
├── Code and automation
│   ├── Actions
│   ├── Deployments
│   ├── Environments
│   └── ...
└── GitHub Pages ← ここ
```

#### ステップ4：デプロイされたページのURLを確認

Pages設定を開くと、以下のような情報が表示されます：

**「Your site is published at」** という表示があれば、その下に表示されているURLが、あなたが作成したページの公開URLです。

例：`https://username.github.io/repository-name/`

### ポイント：初心者が見落としやすい点

- **Pages設定が「有効」になっているか確認**
  - デフォルトでは無効の場合があります
  - 「Source」が「main branch」または「gh-pages branch」に設定されているか確認

- **デプロイに時間がかかる場合がある**
  - GitHub Pagesは自動デプロイされますが、反映に数秒〜数分かかる場合があります
  - 急いで何度もリロードせず、少し待ってみましょう

- **キャッシュをクリアする**
  - ブラウザのキャッシュが邪魔することもあります
  - Ctrl+F5（または Cmd+Shift+R）でハード更新を試してみてください

### 実体験：私の「迷子事件」から学んだこと

正直なところ、私はこの手順を見つけるまでに30分以上かかりました。

「ページがどこにあるのか」という基本的なことすら、初心者にはわかりません。GitHubのUIは高機能ですが、最初は情報が多すぎて、目的の情報を見つけるのに苦労します。

しかし、一度「Settings > Pages」という導線を知ると、次からは迷いません。

## 第2章：Claude API制限問題と解決策

### 問題：朝からAPI制限にかかり作業ができない

本日の朝、私はClaudeを使い始めて間もなく、使用制限にぶち当たりました。

「あれ？動かない」から始まり、気づいたら「今日は使えないのか…」という絶望的な状況に。

開発ツールとして頼りにしていたClaudeが突然使えなくなるのは、特に初心者にとって致命的です。

### 原因：デフォルトモデルの制限

Claudeは複数のモデルを提供しており、デフォルトではより高性能で高機能なモデルが設定されています。

これらのモデルには**使用制限（Rate Limit）**が設定されており、短時間に大量の利用をするとブロックされます。

| モデル | 高性能 | 制限の厳しさ | 向いている用途 |
|--------|--------|--------|--------|
| Claude 3.5 Sonnet | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 複雑な分析、創造的タスク |
| Claude 3 Haiku | ⭐⭐⭐ | ⭐ | 高速処理、軽量タスク |

### 解決方法：Haiku 4.5への軽量化

私の解決策は**Haiku 4.5というより軽量なモデルに切り替える**ことでした。

#### 具体的な切り替え手順

**Claude WebUI（オンライン版）の場合：**

1. Claude公式ページを開く（https://claude.ai）
2. ページ左下の「Settings」をクリック
3. 「Model Selection」または「Default Model」を探す
4. **「Claude 3 Haiku」** または **「Claude 3.5 Haiku」** を選択
5. 「Save」をクリック

**Claude Code（CLI版）の場合：**

1. ターミナルで以下のコマンドを実行：
```bash
claude config --model haiku
```

2. 確認：
```bash
claude config --show
```

#### なぜHaiku 4.5が最高か

切り替え後、私が感じた利点は：

✅ **制限にかかりにくい** - より多くのリクエストを処理できる
✅ **速い** - 応答が素早く、開発のテンポが良くなる
✅ **十分な性能** - 初心者の開発タスクなら全く問題ない性能
✅ **快適さ** - 「蛇蛇に使うことができた」（すばやく連続して使える）

### 重要なポイント

- **Haiku 4.5は「劣化版」ではない**
  - 確かに最高性能モデルと比べると機能は少ないです
  - ですが、99%の初心者タスクなら十分すぎるほど高性能です

- **モデルは後から変更できる**
  - 「Haiku 4.5で足りない」と感じたら、いつでも高性能モデルに戻せます
  - 自分のニーズに合わせて柔軟に選択することが重要です

- **最初から最高性能を求める必要はない**
  - 初心者だからこそ、「使える」状態が優先
  - 制限に引っかかって作業が止まるより、軽量モデルで快適に使う方が生産性が上がります

## 第3章：初心者が学べたこと

本日の2つの課題を通じて、私が学んだ大切なことは3つです。

### 1. ツール理解の重要性

GitHubやClaudeは素晴らしいツールです。

しかし、素晴らしいからこそ、機能が豊富で、初心者には複雑に見えます。

**「わかる」まで時間がかかることは珍しくない**

むしろ、困った時に自分で試行錯誤して解決する経験が、最良の学習になります。

### 2. 問題解決のスキル

「なぜダメなのか」を考え、「どうやって解決するか」を探る。

この過程は、プログラミング本体と同じくらい重要なスキルです。

GitHubの操作に30分困ったことは、一見すると「時間の無駄」に見えるかもしれません。

でも、次からは迷わない知識が身につきました。

### 3. ツールの「最適化」という考え方

「高性能＝良い」ではなく、「自分のニーズに合っている＝良い」

Haiku 4.5への切り替えは、単なる「ダウングレード」ではなく、自分の開発スタイルに合わせた**最適化**です。

この視点は、今後のプログラミング人生で何度も出てくる考え方です。
